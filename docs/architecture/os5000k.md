# OS5000k — In-World Operating System

## Overview

OS5000k is an in-world mini-OS that opens when players interact with computer terminals in Club Mutant. It's a fork of [OS13k](https://github.com/KilledByAPixel/OS13k) (a vanilla JS tiny operating system) extended with a **postMessage bridge** that connects it to the main Club Mutant app — giving OS5000k programs access to player profiles, friends, direct messaging, and playlists.

## PostMessage Bridge Protocol

Three message types flow between the React host and the OS5000k iframe:

| Type | Direction | Purpose |
|------|-----------|---------|
| `os5k:request` | iframe → host | Request with method + payload, expects response |
| `os5k:response` | host → iframe | Response with payload or error, keyed by request id |
| `os5k:push` | host → iframe | Unsolicited event (new mail, presence update, etc.) |

### Request Methods

| Method | Payload | Returns |
|--------|---------|---------|
| `profile.getSelf` | — | `UserProfile` |
| `profile.getUser` | `{ userId }` | `UserProfile` |
| `friends.list` | `{ state? }` | `{ friends: FriendEntry[] }` |
| `friends.add` | `{ userId?, username? }` | `{ success }` |
| `friends.remove` | `{ userId }` | `{ success }` |
| `mail.send` | `{ recipientId, subject, body }` | `{ messageId }` |
| `mail.listConversations` | `{ cursor? }` | `{ conversations, cursor? }` |
| `mail.getMessages` | `{ otherUserId, cursor? }` | `{ messages, cursor? }` |
| `mail.markRead` | `{ otherUserId }` | `{ success }` |
| `playlists.list` | — | `{ playlists: PlaylistEntry[] }` |

### Push Events

| Event | Payload | Source |
|-------|---------|--------|
| `system.connected` | `{ userId, username }` | Sent on iframe load |
| `mail.newMessage` | `MailMessage` | Nakama notification (code 100) |
| `friends.presenceUpdate` | `{ joins, leaves }` | Colyseus presence |

## Bridge Injection for Nested Iframes

OS13k runs each program in its own `<iframe>`. The bridge SDK attaches to `window.OS5000k` at the top OS13k level. When OS13k opens a program, `OS13kWindow.js` injects the bridge into the program's iframe:

```javascript
// OS13kWindow.js (line ~254)
if (window.OS5000k) iframeContent.OS5000k = window.OS5000k;
```

Programs use a `waitForBridge()` polling pattern since the injection happens asynchronously after iframe load.

## Direct Messaging (DM) System

### Storage Design (Nakama Storage Engine)

**Dual-write pattern** — each message stored twice:
- `dm_messages` collection, key = `{timestamp}_{messageId}`
  - Sender copy: `userId = senderId`
  - Recipient copy: `userId = recipientId`

**Conversation index** — secondary collection for efficient inbox listing:
- `dm_conversations` collection, key = `conv_{otherUserId}`
  - Fields: `otherUserId`, `otherUsername`, `lastMessagePreview`, `lastMessageAt`, `unreadCount`
  - Updated atomically on every send/receive

### RPCs (all ES5, registered in `nakama/modules/index.js`)

- `send_message` — validates, rate limits (20/min), dual-writes, updates conversation index, sends notification (code 100)
- `list_conversations` — paginated from `dm_conversations` sorted by `lastMessageAt` desc
- `get_messages` — scans `dm_messages` filtering by conversation partner
- `mark_read` — resets `unreadCount` to 0

## Build System

- **Package**: `@club-mutant/os5000k` in `packages/os5000k/`
- **Build**: `node build.mjs` — copies `static/` to `dist/`, compiles `bridge-sdk.ts` to IIFE with esbuild, injects `<script>` tag into `index.html`
- **Dev**: Vite plugin (`client-3d/vite.config.ts`) serves `dist/` at `/os5000k/` in dev, copies to `client-3d/dist/os5000k/` in production
- **Deployment**: Cloudflare Pages `_redirects` has passthrough rule `/os5000k/* /os5000k/:splat 200`
